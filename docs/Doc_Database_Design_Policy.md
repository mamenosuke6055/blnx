# データベース設計方針

## 方針

**家計と投資を分断せず、すべての金融資産を単一の複式簿記データベース（`finance.db`）で一元管理する。**

1.  **データの一貫性**:
    - すべての取引（現金の移動、カード利用、株式購入、配当金など）が単一のシステムで管理されるため、データの矛盾が生じない。

2.  **正確な資産追跡**:
    - すべての資産・負債・収益・費用の動きを一つの帳簿で正確に追跡できる。

3.  **分析の容易化**:
    - 資産全体のポートフォリオ分析や、投資リターンを含めた家計全体の分析が容易になる。

## 統合データベースで、家計と投資を分けて表示できるか？

**可能である。**

`accounts`テーブル（勘定科目マスタ）の階層構造を利用することで、表示や集計の段階でデータを柔軟に切り分けることができる。

#### 勘定科目の設計例

```
- 資産 (ASSET)
  - 現金・預金 (CASH)
    - 楽天銀行
  - 投資 (INVESTMENT)
    - 証券口座
      - JT株
      - S&P500 ETF
- 費用 (EXPENSE)
  - 家計費
    - 食費
    - 交通費
  - 投資費用
    - 手数料
```

- **家計のキャッシュフロー**: `費用:家計費` 配下の勘定科目に紐づく取引を集計する。
- **投資パフォーマンス**: `資産:投資` 配下の勘定科目に絞って評価額や損益を集計する。
- **期間指定**: `transactions`テーブルの日付を元に、週次や月次での集計も可能。

これにより、データ管理の一元化と、表示・分析の柔軟性を両立する。

## `splits`テーブルの設計思想

現在の`splits`テーブルは、1つの取引（transaction）が複数の明細（split）に分かれて記録される設計を採用しています。これは本格的な複式簿記ソフトウェアで一般的に用いられるデータモデルであり、以下の重要なメリットがあります。

### 1. N対Mの複雑な取引の表現

従来の「借方/貸方」カラムを持つテーブルでは、1対1の単純な取引しか記録できません。しかし、現実の取引は複数の勘定科目が絡むことが頻繁にあります（例：給与から税金や社会保険料が天引きされ、残りが預金に振り込まれる）。

`splits`テーブルの設計では、1つの`transactions.guid`（取引ID）に対して複数の`splits`レコードを紐付けることで、N対Mの複雑な取引を正確に表現できます。

**例：給与の受け取り**

| `tx_guid` (取引ID) | `account_guid` (勘定科目) | `value_num` (金額) | `value_denom` |
| :--- | :--- | :--- | :--- |
| (同じ取引ID) | 普通預金 (資産) | +250,000 | 1 |
| (同じ取引ID) | 所得税 (費用) | +10,000 | 1 |
| (同じ取引ID) | 社会保険料 (費用) | +40,000 | 1 |
| (同じ取引ID) | 給与 (収益) | -300,000 | 1 |
| **合計** | | **0** | |

この方式の唯一のルールは、「**1つの取引に属するすべての`splits`の金額（`value_num`）の合計は、必ずゼロになる**」という点です。これにより、複式簿記の貸借一致の原則がデータベースレベルで保証されます。

### 2. プラス・マイナス値と借方・貸方の関係

`splits`テーブルの`value_num`カラムのプラス・マイナスは、会計上の「借方(Debit)」「貸方(Credit)」と勘定科目の種類（資産、負債、純資産、費用、収益）によって以下のように対応します。

| 勘定タイプ | 金額が増加するとき | 金額が減少するとき | `splits`テーブルでの表現（増加時） |
| :--- | :--- | :--- | :--- |
| **資産 (Assets)** | 借方 (Debit) | 貸方 (Credit) | **プラス (+) の値** |
| **費用 (Expenses)** | 借方 (Debit) | 貸方 (Credit) | **プラス (+) の値** |
| **負債 (Liabilities)** | 貸方 (Credit) | 借方 (Debit) | **マイナス (-) の値** |
| **収益 (Income)** | 貸方 (Credit) | 借方 (Debit) | **マイナス (-) の値** |
| **純資産 (Equity)** | 貸方 (Credit) | 借方 (Debit) | **マイナス (-) の値** |

この設計により、単純な買い物から複雑な給与取引、投資の売買（手数料や税金も含む）まで、あらゆる金融取引を正確かつ柔軟に記録できる基盤が提供されます。

> 符号付き金額（`value_num` の正負）と `side: 'debit' | 'credit'` フィールド方式の関係は
> `double_entry_bookkeeping_principles.md` §2.3 の注記を参照。
