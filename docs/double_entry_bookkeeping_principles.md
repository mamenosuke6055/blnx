# 複式簿記の基本原則 — 家計簿アプリ実装ガイド

このドキュメントは、複式簿記の原則に基づいた家計簿アプリを実装する際の理論的な土台となるリファレンスです。コーディングの前に必ず参照し、ロジックがこの原則に沿っているか確認してください。

---

## 1. 複式簿記の根本原則

### 1.1 取引の二面性 (Duality)

**すべての経済取引は、必ず2つの側面を持つ。**

ひとつの取引は「価値の増加・発生」と「価値の減少・支出」の両面から記録される。家計簿で言えば「給料が振り込まれた」という出来事は、

- 預金 (資産) が増えた
- 給料 (収益) が発生した

という2つの側面を同時に記録する。**片方だけを記録することは絶対にない。**

### 1.2 貸借平均の原理 (Balance Principle)

**すべての仕訳において、借方合計と貸方合計は必ず一致する。**

```
Σ(借方金額) ≡ Σ(貸方金額)
```

この恒等式は仕訳単位でも、ある期間の全仕訳合計でも常に成立する。実装上、これはバリデーションの中核となる不変条件 (invariant) である。

**実装指針:**
- 仕訳 (Transaction / Journal Entry) を保存する前に、必ず借方合計 == 貸方合計をチェックする
- 一致しない場合はトランザクションをロールバックし、エラーを返す
- 金額は浮動小数点ではなく、整数 (最小通貨単位、例: 円なら円単位) または Decimal 型で扱う

### 1.3 会計等式 (Accounting Equation)

複式簿記の根底にある等式:

```
資産 = 負債 + 純資産
```

家計簿では純資産を「正味財産」「自己資本」などと呼ぶこともある。収益と費用を含めた拡張形は:

```
資産 = 負債 + 純資産 + (収益 − 費用)
```

つまり「収益 − 費用」が当期の純利益となり、期末には純資産に振り替えられる。

## 2. 勘定科目の5分類と借方・貸方ルール

### 2.1 5つの勘定区分

すべての勘定科目は以下の5区分のいずれかに属する。

| 区分 | 英語 | 家計簿での例 |
|------|------|-------------|
| 資産 (Assets) | Assets | 現金、普通預金、定期預金、証券、不動産、貸付金 |
| 負債 (Liabilities) | Liabilities | クレジットカード未払金、借入金、住宅ローン、未払費用 |
| 純資産 (Equity) | Equity / Net Assets | 元入金、繰越利益 (純資産剰余) |
| 収益 (Revenue / Income) | Income | 給与収入、副業収入、利息収入、配当金、贈与受取 |
| 費用 (Expenses) | Expenses | 食費、住居費、光熱費、通信費、交通費、娯楽費、税金、保険料 |

### 2.2 借方 (Debit) と貸方 (Credit) の増減ルール

**これは複式簿記の最重要ルール。実装時のあらゆるロジックの基礎となる。**

| 区分 | 増加 | 減少 |
|------|------|------|
| 資産 | 借方 (Debit) | 貸方 (Credit) |
| 費用 | 借方 (Debit) | 貸方 (Credit) |
| 負債 | 貸方 (Credit) | 借方 (Debit) |
| 純資産 | 貸方 (Credit) | 借方 (Debit) |
| 収益 | 貸方 (Credit) | 借方 (Debit) |

**覚え方:** 「資産・費用」は借方が定位置、「負債・純資産・収益」は貸方が定位置。定位置側で増え、反対側で減る。

**実装指針:**
- 勘定科目マスタには「区分 (category)」と「定位置 (normal_balance: debit or credit)」のフィールドを持たせる
- 残高計算ロジックは次の式に従う:
  - 定位置が借方の科目: 残高 = Σ(借方金額) − Σ(貸方金額)
  - 定位置が貸方の科目: 残高 = Σ(貸方金額) − Σ(借方金額)

### 2.3 借方・貸方は「左・右」であり「プラス・マイナス」ではない

借方 (Debit) と貸方 (Credit) は単なる位置情報。日常的な「収入 = プラス、支出 = マイナス」という直感では複式簿記は組み立てられない。**コード上で `is_positive` のようなフラグを設けるのは禁忌。** 必ず `side: 'debit' | 'credit'` で表現すること。

> **blnx の現行実装に関する注記** — blnx は歴史的経緯から、本書の `side` フィールド方式ではなく
> 符号付き金額（`splits.value_num` の正負 = 借方/貸方）を独自規約として採用している（2026-05-13 確定）。
> 本書の side 方式（§3 推奨スキーマを含む）は将来の移行目標形として保持する。いずれの表現でも
> 貸借平均（取引内の合計ゼロ）の検証は必須である。

## 3. 仕訳 (Journal Entry) のデータモデル

### 3.1 推奨スキーマ

仕訳は1つのヘッダ (Transaction) と複数の明細行 (Entry / Posting) で構成する。

```
Transaction (仕訳ヘッダ)
├── id: UUID
├── date: 日付 (取引発生日)
├── description: 摘要 (取引内容)
├── memo: 備考 (任意)
└── entries: Entry[] (2行以上)

Entry (仕訳明細)
├── id: UUID
├── transaction_id: 親への参照
├── account_id: 勘定科目への参照
├── side: 'debit' | 'credit'
└── amount: Decimal (常に正の値)
```

### 3.2 不変条件 (Invariants)

実装上、必ず満たすべき条件:

1. **エントリ数は2以上** — 単式記入は禁止
2. **各エントリの金額は正の値** — 負の金額で「逆仕訳」を表現してはならない (取消は別途逆仕訳で行う)
3. **借方合計 = 貸方合計** — 1円でもズレたら保存させない
4. **少なくとも借方1件・貸方1件以上を含む** — 全部借方や全部貸方はあり得ない

### 3.3 複合仕訳 (Compound Entry)

借方または貸方に複数の科目が登場する仕訳。これも複式簿記の正常な形である。

例) スーパーで食費 3,000円・日用品 1,500円を現金で支払い:

```
借方: 食費   3,000 / 貸方: 現金 4,500
借方: 日用品 1,500 /
```

データモデルが `entries: Entry[]` (可変長) になっているのはこのため。常に2行とは限らない。

## 4. 家計簿でよくある取引パターン

実装時のテストケースとしても使える。

### 4.1 給与の受取

```
借方: 普通預金 250,000 / 貸方: 給与収入 300,000
借方: 所得税    10,000 /
借方: 社会保険料 40,000 /
```

源泉徴収を費用として計上する例。簡略化するなら手取りのみ記帳してもよいが、本格的にやるなら総額主義で記録する。

### 4.2 現金での買い物 (食費の支払い)

```
借方: 食費 1,200 / 貸方: 現金 1,200
```

### 4.3 クレジットカード利用 (利用時)

```
借方: 食費 3,000 / 貸方: 未払金 (クレカ) 3,000
```

費用は発生主義で利用時に計上。支払いは負債 (未払金) として記録。

### 4.4 クレジットカードの引き落とし

```
借方: 未払金 (クレカ) 50,000 / 貸方: 普通預金 50,000
```

このときは費用は発生しない。負債が消えて資産が減るだけ。

### 4.5 銀行口座間の振替

```
借方: 定期預金 100,000 / 貸方: 普通預金 100,000
```

資産から資産への移動。収益・費用は発生しない。

### 4.6 借入金の返済 (元本 + 利息)

```
借方: 借入金 9,000 / 貸方: 普通預金 10,000
借方: 支払利息 1,000 /
```

元本部分は負債の減少、利息部分は費用の発生。

### 4.7 ATM手数料を伴う引出し

```
借方: 現金       9,890 / 貸方: 普通預金 10,000
借方: 支払手数料   110 /
```

## 5. 発生主義と現金主義

### 5.1 発生主義 (Accrual Basis)

**取引が発生した時点で記帳する。** 現金の動きとは独立。

- クレジットカードで買い物 → 利用日に費用計上 (現金はまだ動いていない)
- 給料の未収分 → 期間に対応して計上

複式簿記の本来の姿は発生主義。家計簿でも費用と収益の対応関係を正しく把握できる。

### 5.2 現金主義 (Cash Basis)

**現金 (および預金) が動いたときだけ記帳する。**

簡易な家計簿アプリは現金主義で済ませることが多いが、これだとクレジットカード利用時点で費用が見えず、引き落としタイミングでまとめて費用がドンと立つことになり、月次の支出傾向が歪む。

### 5.3 実装上の推奨

- デフォルトは発生主義
- ユーザー設定で切替可能にしてもよいが、混在させないこと
- クレジットカード、後払い決済 (PayPay後払い、QUICPay等) は発生主義で扱うのが望ましい

## 6. 会計期間と決算

### 6.1 会計期間 (Accounting Period)

家計簿では通常「1月1日〜12月31日」を1期間とすることが多い。ユーザーが任意に区切れるようにしてもよい (例: 給料日基準、4月始まり等)。

### 6.2 損益勘定と純資産への振替

期末には収益・費用勘定の残高を **損益勘定 (Income Summary)** に集約し、その差額を純資産勘定に振り替える。

```
収益勘定 → 損益勘定 (収益はゼロにリセット)
損益勘定 → 費用勘定 (費用はゼロにリセット)
損益勘定の残額 → 繰越利益 (純資産)
```

これにより、次期は収益・費用がゼロから始まる (一時的勘定 / Temporary Accounts)。資産・負債・純資産は次期に繰り越される (恒久的勘定 / Permanent Accounts)。

**実装指針:**
- 期末処理を実行する `closeBooks(periodEnd)` のような関数を用意する
- 期末処理後の収益・費用勘定の残高はゼロになっていることをテストで確認する
- 期末処理自体も通常の仕訳として記録する (監査可能性の確保)

## 7. 財務諸表

複式簿記から自動的に導出できる基本的な財務諸表。

### 7.1 貸借対照表 (Balance Sheet / B/S)

**ある一時点における資産・負債・純資産のスナップショット。**

```
[資産の部]               [負債の部]
 現金        50,000       未払金     80,000
 普通預金   800,000       借入金    500,000
 定期預金 1,000,000     ─────────────────
                         [純資産の部]
                         元入金   1,000,000
                         繰越利益    270,000
─────────────         ─────────────────
資産合計  1,850,000      負債・純資産合計 1,850,000
```

左右の合計は必ず一致する (会計等式の表現)。

### 7.2 損益計算書 (Profit & Loss / P/L)

**一定期間における収益と費用の集計。差額が純損益。**

```
[収益の部]
 給与収入       3,000,000
 副業収入         500,000
─────────────────
 収益合計       3,500,000

[費用の部]
 食費            600,000
 住居費        1,200,000
 ...
─────────────────
 費用合計       3,230,000

当期純利益         270,000
```

### 7.3 試算表 (Trial Balance)

全勘定科目の残高一覧。借方残高合計と貸方残高合計が一致することを確認するための検算表。**実装上、定期的に試算表で貸借一致を検証するロジックがあるとバグ検出に有効。**

## 8. 実装時のチェックリスト

家計簿アプリの仕訳機能を実装する際の最終チェックリスト。

### データモデル
- [ ] 勘定科目に `category` (asset/liability/equity/income/expense) と `normal_balance` (debit/credit) フィールドがある
- [ ] 仕訳は親 (Transaction) と子 (Entry) の1対多構造
- [ ] Entry の金額は常に正の Decimal / 整数
- [ ] Entry には `side: 'debit' | 'credit'` がある

### バリデーション
- [ ] 仕訳保存前に借方合計 == 貸方合計をチェック
- [ ] Entry が2件以上あることをチェック
- [ ] 借方・貸方それぞれ最低1件あることをチェック
- [ ] 金額がゼロまたは負の Entry を弾く

### 取引処理
- [ ] 仕訳保存はデータベーストランザクション内で行う (途中で失敗したら全ロールバック)
- [ ] 仕訳の修正は「逆仕訳 + 新規仕訳」で行う (履歴を残す。ハードな上書き編集は避けるのが望ましい)
- [ ] 削除も論理削除 + 逆仕訳が望ましい

### 計算ロジック
- [ ] 勘定残高は normal_balance に応じて (借方 − 貸方) または (貸方 − 借方)
- [ ] 貸借対照表は資産合計 = 負債合計 + 純資産合計 を検証
- [ ] 損益計算書の当期純利益が、期首・期末純資産の差と一致することを検証

### 期末処理
- [ ] 損益勘定への振替仕訳を生成する関数がある
- [ ] 期末処理後、収益・費用勘定の残高がゼロになる
- [ ] 期末処理も通常の仕訳として記録される

### 浮動小数点の禁止
- [ ] 金額計算に float / double を使っていない
- [ ] Decimal 型または整数 (最小通貨単位) を使っている

## 9. 用語集 (Glossary)

| 用語 | 英語 | 意味 |
|------|------|------|
| 仕訳 | Journal Entry / Transaction | 取引を借方・貸方に分けて記録すること |
| 勘定 | Account | 記録の集計単位 (現金、食費など) |
| 元帳 | Ledger | 勘定ごとの取引履歴 |
| 借方 | Debit (Dr) | 仕訳の左側 |
| 貸方 | Credit (Cr) | 仕訳の右側 |
| 残高 | Balance | ある勘定の現在の残額 |
| 試算表 | Trial Balance | 全勘定の残高一覧 |
| 貸借対照表 | Balance Sheet | 資産・負債・純資産のスナップショット |
| 損益計算書 | Profit & Loss Statement | 収益・費用・純損益の期間集計 |
| 発生主義 | Accrual Basis | 取引発生時に記帳する方式 |
| 現金主義 | Cash Basis | 現金移動時のみ記帳する方式 |
| 決算 | Closing | 期末に帳簿を締める処理 |
| 逆仕訳 | Reversing Entry | 取消や修正のために逆向きに記録する仕訳 |
| 複合仕訳 | Compound Entry | 借方または貸方に複数科目が登場する仕訳 |
